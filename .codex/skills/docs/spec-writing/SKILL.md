# 한국어 Spec 문서 작성

## 개요

- spec 문서 작성 및 수정 시 참고용 규칙
- 작업 목적, 결정 사항, 검증 기준의 빠른 파악 가능 형태 지향

## 적용 조건

- `.codex/spec` 아래 Markdown 문서 신규 작성 시점
- 기존 spec 보완 또는 재구성 시점
- 계획, 설계, 실행 기준 정리 시점

## 언어 규칙

- 설명, 문장, 항목, 문서형 주석 = 모두 한국어
- 코드 예시 내 문법 토큰, 명령어, 식별자, 경로 = 원문 유지 가능
- 코드 블록 내부 주석 = 필요 시 한영 병기 허용
- 코드 블록 외부 설명 = 한국어만 사용
- frontmatter 기술적 키 이름 = 원문 유지 가능
- 불필요한 영어 문장 설명 사용 금지
- 문장 마무리 = 명사형 우선
- 장문 설명보다 개조식 우선

## 작성 원칙

- 목적과 범위의 선노출
- 배경보다 결정 사항의 우선 배치
- 학습용 프로젝트 기준의 이해 가능한 설명 우선
- 구현 세부사항과 이유, 기준의 함께 기재
- 검증 방법 없는 spec = 완료 상태 아님

## 출력

- 자연스러운 한국어 설명
- 한국어 중심의 Markdown 문서
- 재사용 가능한 섹션 구조

## 작성 절차

1. 문서 목적 정리
2. 포함 범위와 제외 범위 분리
3. 핵심 결정과 이유 정리
4. 작업 계획과 검증 방법 작성
5. 미확정 항목의 마지막 분리

## Spec 템플릿

```md
# 제목

## 목적
- 이 문서가 해결하려는 문제 정리

## 범위
- 포함되는 내용
- 제외되는 내용

## 배경
- 작업 필요 이유 정리

## 핵심 결정
- 결정 1
- 결정 이유
- 결정 2
- 결정 이유

## 설계
### 구조
- 주요 구성요소와 역할

### 흐름
- 입력, 처리, 출력 흐름

### 제약 및 예외
- 실패 조건
- 주의할 제약

## 작업 계획
- 작업 1
- 작업 2

## 검증
- 확인 항목
- 실행 명령어 또는 수동 확인 방법

## 미해결 사항
- 아직 결정하지 않은 항목
```

## 체크리스트

- 제목, 목적, 범위의 한국어 명확성
- 코드 블록 바깥 영어 설명 문장 부재 여부
- 핵심 결정과 이유의 동시 기재 여부
- 작업 계획과 검증 방법의 누락 여부
- 미확정 항목의 분리 여부

## 금지 사항

- 템플릿 없는 긴 본문 선작성 금지
- 영어 문단 기반 배경 또는 설계 설명 금지
- 구현 목록만의 단독 기재 금지
- 검증 방법 없는 완료형 문서 작성 금지
